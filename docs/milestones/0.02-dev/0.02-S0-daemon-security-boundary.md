# 0.02-S0 — Daemon Security Boundary

## Objective

Define and validate the local daemon/client security boundary before daemon API
and TypeScript client surfaces stabilize. This sprint strengthens the SDK/API
boundary, work-order prerequisite contract, and trace/audit attribution contract
without adding a daemon server or side-effect path.

## Functional scope

- Adds Rust reference types and fail-closed validation in `splendor-types` for
  caller principal identity, endpoint scopes, tenant/fleet binding, audience
  binding, expiry, revocation, signed work-order metadata, explicit local dev
  mode, and audit attribution.
- Documents the daemon security boundary in `docs/reference/daemon-security-boundary.md`.
- Adds tests for anonymous, missing-scope, wrong-binding, wrong-audience,
  expired, revoked, unsigned-work-order, expired-work-order,
  revoked-work-order, incompatible-work-order, invalid-dev-mode, trace-link,
  gateway-bypass or self-attested gateway completion, redaction, percept
  schema/provenance, fleet-bound credential on tenant endpoint, and insecure
  fallback denial paths.

## Non-goals

- No daemon server, HTTP listener, or runtime auth middleware.
- No OAuth/OIDC server.
- No PKI or fleet mTLS rollout.
- No node bootstrap protocol.
- No remote fleet auth.
- No governance approval workflow.
- No broad runtime permission engine.
- No TypeScript runtime enforcement.
- No side-effect authorization outside the Action Gateway.

## Public contracts changed

- New `splendor-types` exports:
  - `AppPrincipal`, `ClientPrincipal`, `CallerCredential`
  - `CredentialBinding`, `CredentialAudience`, `EndpointScope`
  - `RevocationStatus`, `WorkOrderAuthorization`, `WorkOrderSignature`
  - `AuditAttribution`, `InsecureDevMode`, `LocalTransportBinding`
  - `ClientConnectionPolicy`, `GatewayVerificationState`
  - `DaemonEndpoint`, `DaemonSecurityRequest`, `DaemonSecurityDecision`
  - `DaemonSecurityError`
  - `validate_daemon_request`, `validate_insecure_dev_mode`,
    `validate_client_connection_policy`
- New reference doc: `docs/reference/daemon-security-boundary.md`.

## Trace behavior

No trace event classes are added in this sprint. The boundary requires mutating
daemon calls to carry `AuditAttribution` so future daemon endpoints can persist
caller identity into trace/audit metadata. Action submissions must be trace-linked
and must not bypass gateway verification.

## State behavior

No state nodes are created or modified. Run creation/resume validation is a
precondition before future runtime state changes; rejected requests do not create
runs, resume runs, or advance state.

## Gateway and verifier behavior

The daemon security validator fails closed for missing authentication, missing
scopes, wrong resource binding, wrong audience, expired/revoked credentials,
missing/unsigned/expired/revoked/incompatible work orders, missing attribution,
invalid dev mode, disallowed percepts, missing trace redaction, missing action
trace linkage, and gateway bypass attempts.

At the daemon boundary, action submit accepts only
`GatewayVerificationState::Required`. Self-attested `Completed` and explicit
`Bypassed` markers are denied. Adapter execution conditions are otherwise
unchanged: side effects still require the Action Gateway and verifier chain.

## Replay behavior

The sprint adds no replay executor and no side-effect path. Replay can later
inspect authorization decisions and audit attribution from trace metadata. It
must not reuse caller credentials or work orders to re-execute side effects.

## Failure behavior

All validator failures return structured `DaemonSecurityError` variants and deny
the request. No error converts into an implicit allow.

## Future extension notes

0.02-S5 daemon API work should call `validate_daemon_request()` before creating
or resuming runs, appending percepts, reading traces, or submitting actions.
0.02-S6 clients should map their connection options to `ClientConnectionPolicy`
and refuse silent insecure fallback. 0.03 signed-work-order work can replace
signature metadata presence checks with cryptographic verification while keeping
the same fail-closed contract.
