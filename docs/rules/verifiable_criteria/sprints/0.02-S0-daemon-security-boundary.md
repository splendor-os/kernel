# 0.02-S0 — Daemon Security Boundary: Verifiable Criteria

**Milestone:** Splendor0.02-dev
**Objective:** Define the secure communication boundary between apps, SDKs, CLIs, control planes, adapters, sidecars, and Splendor daemons before daemon APIs and clients stabilize.

**Bounded scope**

- Documentation, reference Rust contract types, and fail-closed validation only.
- App/client caller identity, endpoint scopes, credential binding, signed work-order requirements, audit attribution, and insecure-dev-mode restrictions.
- Prerequisite criteria for later runtime daemon API and TypeScript client work.

**Explicit non-goals**

- No daemon server, network listener, runtime auth middleware, or production credential verifier.
- No full OAuth server.
- No full PKI management.
- No fleet mTLS rollout.
- No node bootstrap protocol.
- No remote fleet auth.
- No governance approval workflow.
- No runtime permission engine beyond documented criteria.

## Required Documentation

The sprint is complete only if the following are documented:

- caller identity model;
- transport modes;
- endpoint scopes;
- tenant/fleet binding;
- audience binding;
- expiry;
- revocation;
- expired or revoked work-order rejection for run creation and resume;
- revocation source expectations, such as a revocation list, introspection endpoint, or signing-key invalidation;
- signed work-order requirement;
- trace/audit attribution;
- insecure dev mode restrictions;
- SDK/client insecure fallback behavior.

## Acceptance Criteria

| ID | Criterion | Pass condition |
|---|---|---|
| VC-0.02-S0-01 | Anonymous daemon calls | Docs state that all non-dev anonymous daemon calls must be rejected. |
| VC-0.02-S0-02 | Local default | Docs state daemon must not expose unauthenticated TCP by default. |
| VC-0.02-S0-03 | Dev insecure mode | Docs require explicit flag, local-only binding, and visible warning. |
| VC-0.02-S0-04 | App identity | Docs define caller identity separately from tenant, agent, run, node, and instance. |
| VC-0.02-S0-05 | Endpoint scopes | Docs require endpoint-level scopes for daemon operations. |
| VC-0.02-S0-06 | Tenant/fleet binding | Docs require credentials to bind to tenant or fleet context. |
| VC-0.02-S0-07 | Audience binding | Docs require credentials to bind to intended daemon, instance, fleet, or manager audience. |
| VC-0.02-S0-08 | Expiry | Docs require credentials and work-order authority to expire. |
| VC-0.02-S0-09 | Revocation | Docs define a revocation path for credentials and work orders. |
| VC-0.02-S0-10 | Run creation | Docs require valid caller identity, required scope, and signed work order for run creation. |
| VC-0.02-S0-11 | Percept append | Docs require run binding and allowed percept provenance/schema for app-submitted percepts. |
| VC-0.02-S0-12 | Trace read | Docs require tenant/run visibility and redaction policy for trace reads. |
| VC-0.02-S0-13 | Action submit | Docs require action requests to remain trace-linked and gateway verified. |
| VC-0.02-S0-14 | Audit attribution | Docs require mutating calls to record caller identity in trace/audit metadata. |
| VC-0.02-S0-15 | SDK behavior | Docs state SDKs must not silently fall back to insecure unauthenticated mode. |

## Required Validation

The reference Rust contract must include negative/fail-closed tests for:

- anonymous non-dev caller;
- missing endpoint scope;
- wrong tenant/fleet binding;
- wrong audience;
- expired credential;
- revoked credential;
- unsigned work order;
- expired work order;
- revoked work order;
- incompatible work order;
- invalid dev-mode binding;
- missing trace/audit attribution for mutating calls;
- app-submitted percept schema/provenance denial;
- trace read without redaction policy;
- action submit without trace linkage or with gateway bypass;
- action submit with self-attested gateway completion from the daemon boundary;
- fleet-bound credential submitted to a tenant-run endpoint;
- SDK/client silent insecure fallback.

## Blocking Conditions

Block completion if the docs imply any of the following:

- a generic daemon token can authorize arbitrary actions;
- run creation is possible without a signed work order;
- side effects can be authorized by caller auth alone;
- SDKs can silently use insecure mode;
- unauthenticated remote TCP is acceptable by default;
- caller identity is confused with agent identity;
- tenant, fleet, run, node, or instance boundaries are overloaded.
